Robotics_Triggers — 整體架構與功能說明

一、這個專案在做什麼？（一句話＋情境）

這個範例示範如何在一段簡單的二維直線運動中，使用 SoftMotion 的觸發機制來「在路徑中的某一點做出反應」，並提早知道觸發點大約會在什麼時候被通過。
你可以把它想成：軸組沿著一條路徑移動，當走到 60% 的位置附近時，PLC 會被「提醒」即將到達觸發點，並在適當的週期內把 triggerFired 設為 TRUE。

二、使用情境與硬體／通訊架構

2.1 任務與程式對應（對照 TaskConfiguration.md）

任務名稱類型週期優先權執行的 POU
MainTaskCyclict#4ms1Main_PRG
SoftMotion_PlanningTaskFreewheelingt#2ms15Planning_PRG
  • MainTask 以 4 ms 週期執行 Main_PRG，負責實際軸組控制與觸發判斷。
  • SoftMotion_PlanningTask 以 Freewheeling 模式執行 Planning_PRG，負責呼叫 SMC_TuneCPKernel() 來設定觸發預測時間。

2.2 軸與裝置結構（對照 DeviceTree_Axes.md）

軸名稱類型綁定任務
a0SM_Drive_Virtual-
a1SM_Drive_Virtual-

這兩個虛擬軸共同組成一個平面軸組（在程式中以 AxisGroup 參考），範例中雖未明確列出軸組設定，但從 Main_PRG 中呼叫 MC_MoveLinearAbsolute() 和 SMC_GroupPrepareTrigger() 可看出它們是以軸組方式運動的。

2.3 Trace 與觀察點（對照 TraceConfig.md）

Trace 記錄了下列變數：

  • Main_PRG.readPosMCS.Position.c.X、Main_PRG.readPosMCS.Position.c.Y：軸組在 MCS（機械座標系）中的當前位置。
  • Main_PRG.readTrigger.TriggerInfo.triggerTime：觸發點距離目前週期開始還剩多少秒。
  • Main_PRG.triggerFired：邏輯層是否已對觸發做出反應。

搭配這些 Trace，可以在示波器中觀察觸發預測與實際動作之間的時序關係。

三、控制流程：由淺入深

3.1 整體流程概觀

  1. Planning_PRG 先在規劃任務中呼叫 SMC_TuneCPKernel()，設定預測時間為 100 ms，並將 initDone 設為 TRUE。
  2. Main_PRG 在 STATE_INIT 等待 Planning_PRG.initDone 完成後，才開始進入實際控制流程。
  3. Main_PRG 先上電並啟用軸組，接著準備一個相對位置為 0.6 的觸發，再指令一段線性移動。
  4. 觸發一旦被成功準備並附加到這段運動上，Main_PRG 透過 SMC_GroupReadTrigger() 讀取觸發資訊。
  5. 當觸發狀態變為 Active 且 triggerTime 小於等於一個任務週期（4 ms）時，代表本週期內就會通過觸發點，Main_PRG 便將 triggerFired 設為 TRUE。

3.2 Main_PRG 狀態機（對照 Main_PRG.st）

Main_PRG 使用 state 與一組常數（STATE_INIT 等）構成狀態機：

  • STATE_INIT (0)：
    等待 Planning_PRG.initDone = TRUE，確保規劃器已設定好預測時間。

  • STATE_POWER_AND_ENABLE (10)：

    • 呼叫 SMC_GroupPower()（變數 power）為整個軸組上電。
    • 呼叫 MC_GroupEnable()（變數 enable）啟用軸組。
    • 啟用成功後，將 readPosMCS.Enable := TRUE，開始持續更新位置資訊。

  • STATE_COMMAND_MOVEMENT_WITH_TRIGGER (20)：

    • 設定 prepTrigger 的 PositionType 為 MvtRelative，相對位置 MvtPosition := 0.6。
    • 呼叫 SMC_GroupPrepareTrigger()，準備觸發並等待 Prepared 成功。
    • 同時設定 moveLinear 的目標位置 (X:=20, Y:=10)，當觸發準備好時才下達運動指令。

  • STATE_ENABLE_READ_TRIGGER (30)：

    • 把 readTrigger.TriggerId 設為 prepTrigger.TriggerId。
    • 將 readTrigger.Enable := TRUE 開啟觸發監控。
    • 接著切換到 STATE_CHECK_TRIGGER。

  • STATE_CHECK_TRIGGER (40)：

    • 持續檢查 readTrigger.TriggerInfo.status 是否為 SMC_TRIGGER_STATUS.Active。
    • 當狀態為 Active 時，readTrigger.TriggerInfo.triggerTime 代表「從本週期開始計算，距離觸發點還有多少秒」。
    • 若 triggerTime <= 0.004（對應 4 ms 的任務週期），代表本週期內會通過觸發點，此時將 triggerFired := TRUE 並切到 STATE_TRIGGER_FIRED。

3.3 規劃任務 Planning_PRG 的角色

Planning_PRG 的程式非常精簡，但角色很關鍵：

  • 呼叫 SMC_TuneCPKernel()，將 fPlanningForecastDuration 設為 0.1 秒。
    這代表 SoftMotion 會盡量在「未來 100 ms 內」提前算出軌跡，讓觸發可被預測。
  • 將 initDone 設為 tuneCpKernel.Done，提供給 Main_PRG 作為啟動條件。

四、各支程式負責哪些功能？（由淺入深）

4.1 Main_PRG.st — 主控制與觸發示範

  • 類型：PROGRAM。
  • 職責：
    • 負責上電、啟用軸組並指令一段直線運動。
    • 準備並附加一個相對位置為 0.6 的觸發到該運動上。
    • 透過 SMC_GroupReadTrigger() 監控觸發狀態與 triggerTime，判斷何時在程式邏輯中設置 triggerFired。

4.2 Planning_PRG.st — 規劃器調校

  • 類型：PROGRAM。
  • 職責：
    • 在規劃任務中呼叫 SMC_TuneCPKernel()，設定預測時間（此處為 100 ms）。
    • 將 initDone 提供給 Main_PRG 作為啟動條件，確保在觸發功能可用之前不會進入實際運動。

五、函式庫與函式／功能塊一覽（用途、引數、回傳）

5.1 函式庫（對照 Libraries.md）

庫名稱解析版本／備註
#SM3_BasicSoftMotion 基礎運動控制功能塊與型態
#SM3_CamBuilder含路徑／凸輪相關功能，本範例主要用到觸發相關 FB

5.2 主要功能塊與函數

  • SMC_GroupPower()（變數 power）

    • 用途：為軸組上電，控制多軸驅動器啟動與調節器啟用。
    • 關鍵引數：AxisGroup、Enable、bDriveStart、bRegulatorOn。

  • MC_GroupEnable()（變數 enable）

    • 用途：啟用軸組，允許後續使用群組運動 FB。
    • 關鍵輸出：Done 表示軸組已就緒。

  • SMC_GroupPrepareTrigger()（變數 prepTrigger）

    • 用途：在指定的軌跡位置上準備一個觸發點，可為相對位置（MvtRelative）。
    • 關鍵引數：
      • PositionType：例如 SMC_TriggerPositionType.MvtRelative。
      • Position.MvtPosition：0.0～1.0 代表運動的起點到終點。
    • 關鍵輸出：Prepared 與 TriggerId。

  • MC_MoveLinearAbsolute()（變數 moveLinear）

    • 用途：在機械座標系 MCS 下執行一段直線絕對移動。
    • 關鍵輸入：AxisGroup、Position、Velocity、Acceleration、Deceleration、Jerk。
    • 本範例用途：作為掛載觸發的實際運動指令。

  • SMC_GroupReadTrigger()（變數 readTrigger）

    • 用途：讀取先前準備的觸發資訊，包括狀態與 triggerTime。
    • 關鍵輸入：TriggerId 需對應到 prepTrigger.TriggerId。

  • MC_GroupReadActualPosition()（變數 readPosMCS）

    • 用途：讀取目前軸組位置，並輸出到 Position 結構中，搭配 Trace 觀察實際位置。

  • SMC_TuneCPKernel()（在 Planning_PRG 中）

    • 用途：設定 SoftMotion 規劃核心的預測時間，讓系統可以提前計算未來的軌跡。
    • 關鍵引數：fPlanningForecastDuration，本範例為 0.1 秒。

七、特別的演算法與觀念

7.1 觸發預測與任務週期的關係

triggerTime 的意義是「從目前這個 MainTask 週期開始往後算，到觸發點還有幾秒」。
當 triggerTime <= BUS_CYCLE_TIME（這裡為 0.004 秒）時，就代表「最多一個週期後一定會通過觸發點」，此時程式就可以把 triggerFired 設為 TRUE，並在接下來的邏輯中採取動作。

7.2 規劃任務與主控制任務分工

這個範例把 SMC_TuneCPKernel() 放在 Planning_PRG 中執行，讓主控制任務 Main_PRG 可以專心處理狀態機與觸發邏輯。
初學者可以把它想成：「規劃任務負責告訴 SoftMotion 要看多遠的未來，主任務則負責根據這些資訊做出控制決策」。

八、重要參數與設定位置（實作時對照）

8.1 任務週期

  • MainTask 週期為 4 ms：
    • 決定了觸發時間判斷閾值 0.004。改變週期時，這個數值也要一併調整。
  • SoftMotion_PlanningTask 週期為 2 ms：
    • 影響規劃器更新頻率，但範例中主要感受是觸發預測的平順程度。

8.2 預測時間（在 Planning_PRG 中）

  • fPlanningForecastDuration := 0.1：
    • 決定最多可以提前 100 ms 得到觸發時間資訊。
    • 若提升此數值，觸發資訊會更早可用，但規劃器負擔也會增加。
