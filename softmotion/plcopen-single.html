PLCopenSingle — 整體架構與功能說明

一、這個專案在做什麼？（一句話＋情境）

這個範例示範如何使用 MC_Power 與 MC_MoveAbsolute，在一根虛擬軸 Drive 上反覆執行「移動到目標位置再回到原點」的往返絕對定位動作，適合初學者練習最基本的單軸定位控制。

在實務上，可以把這個範例想成「只有一個軸的簡單點到點移動」，例如：小型平台在 0 與 100mm 之間來回移動。

二、使用情境與硬體／通訊架構

2.1 硬體與軸配置

對照 DeviceTree_Axes.md，本範例只包含一根虛擬軸：

軸名稱類型綁定任務
DriveSM_Drive_Virtual-

這代表程式是在「軟體模擬軸」上練習運動控制指令，實際連線到實體驅動器前，可以先用這種虛擬軸驗證程式流程是否正確。

2.2 任務與週期

根據 TaskConfiguration.md，專案只有一個主要任務：

任務名稱類型週期優先權POU 清單
TaskCyclict#4ms1MOTION_PRG
  • 任務 Task 每 4ms 週期性執行一次 MOTION_PRG，負責驅動整個運動流程。
  • 由於只有一個任務與一個 POU，控制結構非常單純，適合用來觀察狀態機邏輯與功能塊行為。

2.3 資料流與角色分工（誰控制誰）

  • 任務 Task 週期性呼叫程式 MOTION_PRG。
  • MOTION_PRG 內部透過 MC_Power 與 MC_MoveAbsolute 控制軸 Drive。
  • 軸 Drive 將指令轉換成實際（或模擬）運動。

三、控制流程：由淺入深

3.1 整體流程概觀

整體流程可以用下面幾句話理解：

  1. 一開始，程式先替軸 Drive 「上電並啟動」。
  2. 軸準備好之後，執行一次絕對定位，從目前位置移動到目標位置 p（預設 100）。
  3. 抵達目標位置後，再執行第二次絕對定位，從 p 回到 0。
  4. 接著重複步驟 2 與 3，在 0 與 p 之間來回移動。

這個流程完全由變數 iStatus 控制，是一個經典的「狀態機」寫法。

3.2 狀態機細部說明（iStatus）

在 MOTION_PRG.st 中，主要控制流程寫成：

  • 狀態 0：上電與啟動軸

    • 呼叫 MC_Power()，把 Enable、bRegulatorOn、bDriveStart 都設為 TRUE。
    • 當 Power.Status 為 TRUE 時，代表軸已經上電完成，將 iStatus 改為 1。

  • 狀態 1：從目前位置移到 p

    • 呼叫 MC_MoveAbsolute()，參數 Position := p，Velocity/Acceleration/Deceleration := 100。
    • 一直維持 Execute := TRUE，直到 MoveAbsolute.Done 變成 TRUE。
    • 達到目標位置後，先以 Execute := FALSE 結束這次命令，再把 iStatus 改為 2。

  • 狀態 2：從 p 移回 0

    • 再次呼叫 MC_MoveAbsolute()，這次 Position := 0。
    • 同樣等待 MoveAbsolute.Done，完成後以 Execute := FALSE 結束命令。
    • 把 iStatus 改回 1，形成「1 → 2 → 1 → 2」的循環。

初學者可以把 iStatus 想成「流程目前進行到第幾步」，每個數字代表一個階段。

3.3 與實體機台的對應想像（進階小節）

雖然本範例使用的是 SM_Drive_Virtual，但在實體應用中，你可以把 Drive 想像成：

  • 一根直線平台：在 0mm 與 100mm 之間往返。
  • 一支旋轉軸：在 0 度與 100 度之間反覆定位。

了解這個簡單單軸範例後，再把同樣的寫法擴展到多軸或更複雜的狀態機，就會容易許多。

四、各支程式負責哪些功能？（由淺入深）

本範例只有一支程式 MOTION_PRG.st，沒有額外的 GVL 或其他 POU，因此閱讀負擔很小。

4.1 MOTION_PRG.st

  • 類型：PROGRAM
  • 主要職責：
    • 以 iStatus 為核心狀態機，依序完成「上電 → 移到目標位置 → 移回原點」三個階段。
    • 透過 MC_Power 啟動軸 Drive，再用 MC_MoveAbsolute 發出絕對定位指令。
  • 與硬體／通訊的關係：
    • 直接操作 Device 樹中的虛擬軸 Drive。
    • 若改成實體軸，只要在 CODESYS 中把 Drive 綁定到實際驅動器即可，程式寫法基本相同。

建議閱讀 MOTION_PRG.st 時，一邊看 iStatus 的值如何改變，一邊對照 CASE 區塊內各個狀態的敘述。

5.2 功能塊與主要引數

以下僅列出在本範例程式中實際使用到的功能塊與重要引數，方便對照程式學習。

5.2.1 MC_Power()

  • 用途：
    • 對指定軸上電、啟用調節器、啟動驅動器，使其可以接受運動指令。
  • 主要引數：
    • Axis : AXIS_REF — 要控制的軸（此處為 Drive）。
    • Enable : BOOL — TRUE 時請求上電。
    • bRegulatorOn : BOOL — TRUE 時啟用位置/速度控制器。
    • bDriveStart : BOOL — TRUE 時啟動驅動器。
  • 重要輸出：
    • Status : BOOL — TRUE 代表軸已準備就緒。
  • 回傳型態：VOID（功能塊型式，無直接回傳值）。

5.2.2 MC_MoveAbsolute()

  • 用途：
    • 讓軸以指定的速度與加減速度，移動到某個絕對位置。
  • 主要引數：
    • Axis : AXIS_REF — 目標軸（此處為 Drive）。
    • Execute : BOOL — 上升緣觸發一次新的移動命令。
    • Position : LREAL — 目標位置（程式中分別給 p 與 0）。
    • Velocity : LREAL — 最大運動速度。
    • Acceleration / Deceleration : LREAL — 加速度與減速度。
  • 重要輸出：
    • Done : BOOL — 達到目標位置且命令完成。
    • Busy : BOOL — 命令執行中。
  • 回傳型態：VOID。

本範例沒有獨立的 GVL 檔案，所有變數都在 MOTION_PRG.st 中宣告，也未使用 RETAIN / PERSISTENT。

七、特別的演算法與觀念

7.1 簡單狀態機寫法

這個範例最重要的觀念是：用一個整數變數 iStatus 搭配 CASE 敘述，就可以把整個流程拆成幾個清楚的階段。

你可以這樣理解：

  • 每一個狀態是「流程的一小步」，例如「上電」、「移到目標」、「移回原點」。
  • 在每個狀態裡，只做一件事情，完成後就把 iStatus 改成下一個數字。
  • 這樣一來，即使之後流程變長，只要再新增幾個狀態即可，程式仍然容易閱讀與維護。

7.2 功能塊呼叫與 Done 訊號

在使用 MC_MoveAbsolute() 時，有兩個關鍵點：

  1. 以 Execute := TRUE 啟動命令，並持續呼叫，直到 Done = TRUE。
  2. 命令完成後，記得再呼叫一次並把 Execute := FALSE，讓功能塊「收尾」，避免下次命令無法正確觸發。

這種模式在其他運動控制功能塊上也很常見，可以把它當成共通的操作樣式。

八、重要參數與設定位置（實作時對照）

8.1 軸與軸組

本範例沒有軸組，僅有一根軸：

名稱類型設定位置
DriveSM_Drive_Virtual在 Device 樹中建立虛擬軸，並命名為 Drive；程式中直接以 Axis := Drive 連接。

8.2 任務週期與運動參數

  • 任務週期：在 TaskConfiguration.md 中，Task 週期為 t#4ms，代表每 4ms 呼叫一次程式。
  • 運動參數：
    • p 預設為 100，可在 MOTION_PRG.st 中直接修改。
    • MC_MoveAbsolute 的 Velocity/Acceleration/Deceleration 都設定為 100，若想觀察不同運動表現，可以調整這三個數值。

在實務應用中，通常會把這些參數放到 GVL 或結構體中集中管理，這個範例則用最直接的寫法幫助理解。

九、建議閱讀與修改順序（給初學者）

  1. 先看 MOTION_PRG.st 開頭的變數宣告
    • 弄清楚 iStatus、Power、MoveAbsolute、p 各自代表什麼。
  2. 接著閱讀 CASE iStatus OF 區塊
    • 逐一對照狀態 0、1、2 內的程式，並在紙上畫出流程箭頭。
  3. 最後再回頭對照本文件第三章與第五章
    • 把 MC_Power、MC_MoveAbsolute 的引數與 Done 行為串起來理解。

當你能夠清楚說出這個程式「在什麼條件下從哪個狀態跳到哪個狀態」，就代表已經掌握這個範例的核心。
