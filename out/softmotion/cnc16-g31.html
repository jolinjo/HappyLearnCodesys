CNC16_G31 — 整體架構與功能說明

一、這個專案在做什麼？

本範例對應 G-code 中的 G31 探針指令：軸朝目標位置移動，一旦探針或感測器觸發便立即停止，並將「實際停止位置」回寫到 CNC 解譯器，以便後續程式能以此為新起點繼續加工。核心觀念是結合探針動作、Quick Stop 與 SMC_SetInterpreterStartPosition，完成「清除剩餘距離」與起點重定位。

注意：本專案的 ST 程式僅展示 Path 端 piStartPosition 的使用與主程式變數宣告，實際探針觸發、Quick Stop 與 bAcknProbe 等邏輯大多封裝在庫與預建 FB 裡。

  • PathTask（Path）：較低優先權，以 20 ms 週期讀取 G-code 並產生路徑，且會使用 Main.setStart.StartPos 當作解譯起點。

三、控制流程：由淺入深

3.1 PathTask：讀檔與解譯，使用新起點

Path 程式（Path.st）流程：

  1. 以 SMC_ReadNCFile2 read 讀取 Application/gcode.cnc，bExecute := TRUE。
  2. 以 SMC_NCInterpreter inter 解譯 read.sentences，設定：
    • nSizeOutQueue := SIZEOF(bufIpo)；
    • pbyBufferOutQueue := ADR(bufIpo)；
    • 最關鍵的是 piStartPosition := Main.setStart.StartPos。
  3. 以 SMC_CheckVelocities cv 對 inter.poqDataOut 做速度檢查，輸出 cv.poqDataOut 給插補器。

透過 piStartPosition 的指定，解譯器會將 G-code 中的路徑起點對齊到 Main.setStart.StartPos，也就是由 BusTask 依探針實際停止位置設定的新起點。

3.2 BusTask：探針、Quick Stop 與設定新起點（概念）

在 Main 程式（Main.st）中可以看到：

  • 輸出變數 setStart : SMC_SetInterpreterStartPosition。
  • 軸電源 FB pwX / pwY。
  • 插補器 ipo : SMC_Interpolator。
  • 軸位置控制 capX / capY。
  • 以及座標轉換 FB trafo / trafoF。

雖然原始匯出檔沒有顯示 Main 的完整實作，但根據官方說明與變數命名，可以推知流程概念：

  1. 軸沿某方向朝目標 G-code 所指定的位置移動。
  2. 一旦探針輸入（或 G31 條件）觸發，執行 Quick Stop／停止當前插補。
  3. 以 SMC_SetInterpreterStartPosition 將 StartPos 設為實際停止點，並透過 bAcknProbe 之類信號確認。
  4. PathTask 在下一輪解譯時使用新的 StartPos 作為 G-code 起點，之後程式繼續執行時就以「實際測得的點」為準。

Trace 設定中記錄了 Main.ipo.istatus、piSetPosition、bAcknProbe、bQuick_Stop 等變數，方便觀察探針過程。

四、各支程式負責哪些功能？

4.1 Main.st

  • 類型：PROGRAM。
  • 職責（依變數推估）：
    • 上電與驅動 DriveX、DriveY。
    • 控制插補器 ipo 依 PathTask 輸出路徑執行運動。
    • 在探針動作發生時觸發 Quick Stop，並透過 SMC_SetInterpreterStartPosition setStart 設定新的起點 StartPos。
    • 使用 capX / capY 與 trafo / trafoF 將插補位置轉成各軸指令。

4.2 Path.st

  • 類型：PROGRAM。
  • 職責：
    • 從 Application/gcode.cnc 讀取 G-code。
    • 使用 Main.setStart.StartPos 作為解譯起點，將 G-code 轉成路徑佇列。
    • 將路徑送入 SMC_CheckVelocities 做速度檢查後提供給插補器。

七、特別的演算法與觀念

7.1 探針與「清除剩餘距離」

實務中，G31 探針流程大致如下：

  1. CNC 程式要求軸向某方向移動到目標點（例如 Z-100）。
  2. 若在途中探針觸發，則馬上 Quick Stop，在當前實際位置停止。
  3. 將此實際位置寫入 SMC_SetInterpreterStartPosition.StartPos，並讓插補器與解譯器知道「程式中這一段剩下的距離應該被清除」。
  4. 後續 G-code 再次啟動時，會以這個新位置為起點繼續走下一段程式。

CNC16_G31 範例展示的是第 3～4 步：如何把新的座標回寫到 Interpreter，並在 PathTask 下一次解譯時套用。

7.2 BusTask 與 PathTask 的職責分離

  • BusTask（Main）：
    • 反應速度要求高（t#4ms、優先權 0）。
    • 負責與軸、探針訊號、Quick Stop 等密切互動。
  • PathTask（Path）：
    • 週期較長（t#20ms）、優先權較低。
    • 只需關注 G-code 句子的讀取與解譯。

這樣的分離可以確保探針與停機邏輯足夠即時，而不會被較慢的解譯流程拖累。

九、建議閱讀與修改順序

  1. 先看 TaskConfiguration.md 與 DeviceTree_Axes.md，理解兩任務與兩軸的角色分配。
  2. 再看 Path.st，掌握 piStartPosition := Main.setStart.StartPos 的使用方式。
  3. 接著閱讀 Main.st，對照 TraceConfig 中的變數，理解探針與 Quick Stop 在整體中的位置（即使程式本體在匯出檔中不可見，也能從變數與 FB 名稱推敲流程）。
  4. 若實作自己的 G31 流程，可在 Main 裡擴充實際的探針輸入邏輯與 Quick Stop 條件，並將實際停止點寫入 setStart.StartPos。
